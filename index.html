分布式锁实现

2017-05-23


在分布式应用当中少不了分布式锁，这里讲一讲用zookeeper和redis实现。

zookeeper

非公平锁
1、所有客户端尝试创建’/lock’临时子节点，zk会保证最终只有一个客户端创建成功，那么认为该客户端获取了锁。其他没有获取到的客户端监听该节点的变更情况。
2、若获得该锁的客户端断开连接或者主动去删除该节点，则zk会通知所有监听该子节点变更的客户端，这些客户端收到变更事件以后去发起获取锁的过程（重复获取锁过程）

这种最大的问题是如果并发过多，注册的watcher太多，释放时会通知所有客户端去抢占，性能上不是很友好。会有’羊群效应’。

公平锁
公平锁与非公平锁最大的区别在于在第一步是创建临时顺序子节点，创建后调用getChildren接口获取所有子节点列表，判断自己是否是所有子节点中序号最小的，如果是则获得锁，如果不是，则去监听比自己序号小的那个节点的变更情况即可。

项目中一般不自己去实现，用现成的工具。如在curator包里的【InterProcessMutex】
以上只是简单的思想，还有更复杂的读写锁实现等。

2、获取锁的客户端用del命令删除进行释放。
这种最大的问题在于如果客户端线程被kill或者其他原因挂掉锁不会被释放。造成死锁。

Mysql InnoDB 锁机制

2017-02-28

一些概念

MVCC (Multi-Version Concurrency Control) 好处：读不加锁，读写不冲突，增加系统并发效率。

  • 读操作分为快照读和当前读
  • 锁分为S锁(共享锁)和X锁(排它锁）
  • 隔离级别：
    • Read uncommited 无应用场景。
    • Read commited(RC) 读已提交，oracle默认级别。不可重复读，存在幻读。
    • Repeatable Read(RR) 可重复读，不存在幻读，mysql默认级别。
    • Serializable 性能影响太大，基本不使用。
  • 锁分类：
    • Record lock 行锁
    • Gap lock 范围锁
    • Next-key lock（行锁+范围锁）

RC和RR的快照读不同

RC的快照读是读取最新一个快照的数据，所以在一个事务里如果多次读同一行，可能其他读到其他事务修改过后的数据。时间顺序如下

时间 会话1 会话2
1 事务开始
2 select * from table where id =1；
3 事务开始
4 update table set id=3 where id=1;
5 select * from table where id =1；
6 事务提交
7 select * from table where id =1；
8 事务提交

对于RC来说，2、5读到一条记录，7则读到0条记录。因为读的都是最新的快照
对于RR来说，2，5，7都是读到同一条记录。因为读的都是事务开始时的快照

解决幻读

InnoDB存储引擎在RR隔离级别已经避免了幻读，而Oracle只有在SERIALIZABLE才能解决。
现有表a,数据1，3，6

时间 会话1 会话2
1 事务开始
2 select * from a where id>3 for update
3 事务开始
4 insert into a select 4;
5 事务提交
6 select * from a where id>3 for update

对于RC来说，次序2查出一条记录。次序6查出2两条记录。
对于RR来说，次序2查出一条记录。次序6同样只有一条记录，但是会话B会等待，直到会话1的事务提交（或者超时，曾发现线上有不少超时，分析后是这种情况）

RC 只会对6这条记录加X锁。
RR 在对6这条记录加X锁的同时，在（3，+∞)加了X锁（Gap lock）

RR和RC的一些加锁分析

如何加锁主要和隔离级别和索引类型有关。组合很多，见：何老师的博客【MySQL 加锁处理分析】

参考

  • MySQL 加锁处理分析
  • 《MySQL技术内幕 InnoDB存储引擎第2版》

zookeeper学习

2017-01-24

应用

  • 数据发布/订阅
    适用于配置文件等数据量小、动态变化、各机器共享配置一致的场景。
    将配置数据写在某个数据节点中、其他服务器用watcher监听。

  • 负载均衡
    动态DNS

  • 服务命名

  • 分布式协调/通知

  • 集群管理

  • Master选举
    方式：抢占临时节点、抢不到的注册监听

  • 分布式锁
    • 排他锁 方式：抢占创建临时节点，抢不到的注册监听
    • 共享锁 方式：写临时顺序节点、比较。
  • 分布式队列

数据结构

事务ID ZXID 64位数字 全局唯一 能识别执行顺序。

节点

ZNode、最小单元。

  • 持久节点 持久存在，除非主动删除。
  • 持久顺序节点 持久存在、并且有节点创建先后顺序。
  • 临时节点 临时存在，客户端断开则删除（非TCP连接断开）、只能作为叶子节点。
  • 临时顺序节点 同上、添加了顺序特性。

节点状态信息 Stat

版本

version 当前数据节点数据内容的版本号（操作但若内容不变仍旧会增加版本号）
cversion 当前数据节点子节点的版本号
aversion 当前数据节点acl变更版本号

通过版本号来保证数据原子性操作。乐观锁-CAS实现。更新前比较当前版本和预期版本是否一致。

Watcher 数据变更的通知

watchedEvent 逻辑事务。
watcherEvent 实现了序列化接口 用于网络传输。

当节点的数据发生变更、服务端发送给客户端一个节点数据内容变更事件

变更前后的数据内容需要客户端主动去重新获取数据。

ACL 保障数据的安全

权限模式：Scheme

  • IP 支持网段
  • Digest username：password

  • Super 超级用户

权限： Permission

  • CREATE(C)
  • DELETE(D)
  • READ(R)

学习正则表达式

2017-01-17

  • 量词
    • 【*】 任意次，包括0
    • 【+】 至少1次。1+
    • 【?】 1次或者0次
    • {n} n次
    • {n,} n+次
    • {n,m} n到m次

分组

  • （）-> 捕获分组（capturing group）。后向引用（backreference），从1开始计数（区别于程序语言）
  • 命名分组 （？’name’…） or (?…)
  • 非捕获分组（？:…） 不放入内存，提高性能，后续不适用。 （?i:the）or (?i)(?:the) i标示忽略大小写模式。
  • 原子分组 (?>the) 什么作用？？

字符组

  • []， 又称方括号表达式。
  • [^… ] 取反，不想匹配

贪心、懒惰和占有

量词自身是贪心的。贪心的量词会首先匹配整个字符串。尝试匹配时，它会选定尽 可能多的内容，也就是整个输入。量词首次尝试匹配整个字符串，如果失败则回退 一个字符后再次尝试。这个过程叫做回溯(backtracking)。它会每次回退一个字符， 直到找到匹配的内容或者没有字符可尝试为止。此外，它还记录所有的行为，因此 相较另两种方式它对资源的消耗最大。它先“吃”尽所有的字符，然后每次“吐” 出一点，慢慢咀嚼消化…

懒惰(有时也说勉强)的量词则使用另一种策略。它从目标的起始位置开始尝试寻 找匹配，每次检查字符串的一个字符，寻找它要匹配的内容。最后，它会尝试匹配 整个字符串。要使一个量词成为懒惰的，必须在普通量词后添加一个问号(?)。它 每次只“吃”一点。

占有量词会覆盖整个目标然后尝试寻找匹配内容，但它只尝试一次，不会回溯。占 有量词就是在普通量词之后添加一个加号(+)。它不“咀嚼”而是直接“吞咽”， 然后才想知道“吃”的是什么。

环视

环视是一种非捕获分组，它根据某个模式之前或之后的内容匹配其他模式。环视也 称为零宽度断言。
包括：

正前瞻

ancyent 后面跟marinere，只匹配前面部分。

反前瞻

正前瞻取反，即 ancyent后面不跟marinere，只匹配前面部分

正后顾

匹配marinere，前面为ancyent

反后顾

正后顾取反，匹配marinere，前面不为ancyent

一致性hash算法

2016-05-18

在学习分布式的时候总会碰到一致性hash算法，今天去学习了一下。

场景及要解决的问题

分布式。假设有N台服务器，需要把一个object映射到不同的服务器上。
最开始的方法：hash(object)%N。这样就可以映射到对应的服务器上。
但是这样存在一个很严重的问题： 新增或者删除服务器的时候，N值变了，映射关系乱了，大部分数据就不在原有的服务器上了！

环形hash空间

现在有3台服务器和4个object对象。
1、对三台服务器进行hash（一般是ip地址）。获得三个值Node A、 Node B、Node C
2、对4个object对象进行hash。获得value 1、value 2、value 3、value 4
3、每个value1关联到顺时针遇到的第一个Node。
如图，则
value1—>Node B,
value2—>Node C value3、
value4—>Node A

这样删除或者新增一个服务器则影响的范围不会很大，但是仍然有一个问题，就是分布不够均衡。这里引入了虚拟节点来处理。假设一个node有100个虚拟节点，先将value映射到virtual node,然后再映射到node。图就不画了，自己脑补。虚拟节点的计算：hash(node.ip #1) ,hash(node.ip #2)。
